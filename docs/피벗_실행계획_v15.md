# FarmFi 피벗 실행계획 (v18 기준)

본 문서는 `docs/FarmFi_전체서비스_기획안_v18.md`(유일 정답 문서)를 기준으로, 현재 코드(`frontend/`, `contracts/`, `iot-mock/`)를 STO 투자 플랫폼에서 v18 "공실전환 창업 지원 인프라"로 전환하기 위한 실행계획이다. 모든 판정은 실제 파일경로에 근거하며, 미확인 항목은 "추정"으로 표기한다. 이번 산출물은 격차분석 + 실행계획까지이며 실제 삭제/수정은 포함하지 않는다.

전제(확정 스코프): 블록체인/STO/투자자 레이어(투자·토큰·청약·배당·에스크로·마일스톤 검증·온체인 앵커·DID/VC)는 전부 제거 대상이다. 수익은 두 주머니뿐 — FarmFi 인프라 수익(도입비/운영관리 패키지/성과 리포트/설비 연결 수수료, 기관 지불)과 운영자 매장 직접판매(FarmFi 미경유)다.

## 0. 확정 설계 결정

아래는 실행 착수 전 확정된 결정이며, 본 문서의 나머지 절은 이 전제 위에서 서술된다.

| # | 항목 | 결정 |
|---|---|---|
| 1 | 대체 인증 방식 | **이메일 + 비밀번호**. jose 세션(JWT) 레이어 재사용, `User`에 `passwordHash` 신규, 비밀번호 해시 라이브러리(bcrypt/argon2) 도입. 로그인/회원가입 서버 엔드포인트 신규 |
| 2 | admin(관리자) 페이지 | **존치** + 본문을 기관 성과 리포트·운영 현황 뷰로 개조 (v18 ★핵심 '기관 성과 리포트'의 자연스러운 뷰) |
| 3 | `ai-vision.ts` | **제거** (마일스톤 증빙 OCR 전용, 공간진단 재배치 안 함 — 스코프 단순화) |
| 4 | 판매데이터 입력 | **운영자 수기입력 API**(`api/sales` POST) + 시드 목데이터. 외부 POS 연동 안 함 |
| 5 | `Project` 모델 | **모델명 유지**(광범위 참조로 churn 최소화), 도메인·UI 표기만 '지점' |
| 6 | 스코프 밖 문서 | `docs/pivot-plan.md` **제거**(사업계획서 v4 기반, DID/VC·앵커링이 v18 스코프와 충돌 — 본 문서가 대체). `SAMPLE_README1.md` **유지**(해커톤 제출 템플릿, 무관) |

### 3-2. 인증 레이어 재설계(선행 필수)
현재 로그인 전체가 SIWE(지갑 서명) 기반이라, wagmi/siwe 제거 = 로그인·회원가입 붕괴. 단순 삭제가 아니라 **대체 인증(이메일/기관 계정 등)으로 재작성이 선행**돼야 함.
- `frontend/src/lib/auth.ts` — **유지·개조**. signSession/getServerSession/requireRole 골격 유지. `SessionPayload.walletAddress`(L8,30,71)를 도메인 식별자로 교체. `Role` 타입에서 `investor` 제거
- `frontend/src/lib/useAuth.ts` — **개조**. SIWE 서명 로직(useAccount/useSignMessage/siwe)을 대체 인증으로 재작성. 세션 useQuery/logout/updateRole은 재사용
- `frontend/src/app/providers.tsx` — **개조**. WagmiProvider/RainbowKitProvider만 벗기고 **QueryClientProvider는 유지**(useAuth가 react-query 의존, `useAuth.ts:4,80`). 통째 삭제 불가
- `frontend/src/app/api/auth/me/route.ts` — **개조**. include의 `user.tokenHoldings`(L22,42)와 응답의 walletAddress·identityVerified·realName·investorAnnualLimit·businessRegNo(L36-41) 제거. `ASSIGNABLE_ROLES`(L62)에서 investor 제거하고 operator/landlord/기관(admin)으로 재정의
- 대체 인증 엔드포인트 신규(`api/auth/login`·`api/auth/signup` 가칭) — 이메일+비밀번호로 검증 후 jose 세션 발급. `User.passwordHash` 신규 + bcrypt/argon2 도입

## 4. 신규 구축 목록 (★핵심 3기능 — 전량 신규)

검증 결과 ★핵심 3기능 모두 코드상 실질 구현 [없음] 확정. 생성 클라이언트의 HarvestRecord·ESGReport 등은 참고 자산으로만 활용(VC·BlockchainAnchor는 스코프상 채택 금지).

### 4-1. 재고-생육 연동 "오늘 할 일" (★핵심)
- 모델: `Inventory`(품목·재배중 수량·수확예정) + `Product`(품목 마스터) + `HarvestRecord`(수확량) 신규
- 작업목록: `TaskItem`(가칭) — taskType(watering/inspection/planting/harvest 등)만 담고 **verificationStatus·작업검증 필드는 금지**(§10-5 위반). '오늘 할 일' 표시는 §10-1 재고-생육 연동으로 허용되나 작업 지시·근태·검증은 제외
- API: `api/tasks/today`(가칭) — 재고 수량 + IotData 생육 단계 결합해 수확/보충 대상 산출
- 페이지: `app/today-tasks`(가칭) 또는 DashboardShell '수확·출하 관리' 메뉴 실체화

### 4-2. 판매-재배 연동 (★핵심)
- 모델: `SalesRecord`(가칭) — 품목별 판매량·기간(Transaction.type='revenue'는 STO 원장이라 부적합)
- API: `api/sales`(입력) + `api/sales/trend`(품목별 추이 → 재배 조정 근거 집계) 신규
- 페이지: `app/sales-planting`(가칭) 또는 '판매·출하 분석' 메뉴 실체화
- 입력: 운영자 **수기입력 API**(`api/sales` POST) + 시드 목데이터. 외부 POS 연동 안 함

### 4-3. 기관 성과 리포트 (★핵심, FarmFi 수익원)
- 모델: `Institution`(기관) + `PerformanceReport`/`ESGReport`(가칭) — 공간활용률·생산량·판매·운영현황 월/분기 집계
- API: `api/reports/institution`(가칭) — Space 활용률 + HarvestRecord 생산량 + SalesRecord 판매 + IotData 운영현황 종합
- 페이지: `app/institution-report`(가칭)

### 4-4. 생육 이상 알림 발송(보완 신규)
현재 detect-anomaly·iot/generate는 isAnomaly를 계산·저장만 하고 발송 로직 없음. admin/notify(제거) 제거 후 발송 경로 완전 공백 → 신규 구축 필요. Notification 모델은 유지·개조(4-5)로 그릇 재활용.

### 4-5. 운영자 매칭 확정 저장부(보완 신규)
OperatorApplication에 매칭 결과(어느 Space에 어느 operator)를 담는 관계가 없음 → 1공간 1운영자 배정 저장 모델/필드(`OperatorAssignment` 가칭, 생성 클라이언트에 타입 존재) 신규 필요.

## 7. 리스크·주의

- **Prisma 7 driver adapter 주입 필수**: `new PrismaClient()` 무인자 호출 금지. `PrismaPg` 어댑터를 주입해야 함(`frontend/src/lib/db.ts` 참고). datasource url은 schema가 아니라 `prisma.config.ts`에 있음. 신규 모델 추가 시에도 이 규약 유지.
- **Prisma 역방향 관계 동시 편집**: 제거 모델을 지우면서 User(L27-32)·Project(L98-105)·Notification 필수 FK(L219)를 함께 수정하지 않으면 `prisma validate/generate` 실패. 한쪽만 제거 불가(검증 실증).
- **jose 제거 금지**: 스코프 지시가 제거로 적었으나 `lib/auth.ts`는 유지 대상 17개 라우트의 세션 레이어. 제거 시 인증 전면 붕괴. SessionPayload.walletAddress 필드만 교체하는 개조로 처리.
- **제거 시 깨질 import 연쇄**:
  - `nav-calculator.ts`는 유지/개조 대상 `api/dashboard/[projectId]/route.ts:3,67`에 물려 있음 → dashboard route의 NAV 참조를 **먼저** 걷어낸 뒤 삭제.
  - `providers.tsx`는 통째 삭제 불가 → WagmiProvider/RainbowKit만 벗기고 QueryClientProvider 보존(useAuth react-query 의존).
