# 검증 명세서 (Verification Spec)

FarmFi 마일스톤 자금집행의 **검증 레이어** 명세. 무엇을 어떤 신호로 판정하고, 어떤 임계로 통과/차단하며, 통과 시 온체인에서 어떻게 집행되는지를 정의한다.

> 종류: 레퍼런스 (현재 구현 상태). 코드가 바뀌면 이 문서를 제자리 갱신한다.
> 관련: 마일스톤·정산 수치는 `plan.md`, API 입출력은 `api-spec.md`.

---

## 1. 목적 — 검증 명제 ①②③

투자금은 운영자에게 한 번에 지급되지 않고, 4개 마일스톤 달성 시마다 트랜치(부분)로 풀린다. 이 집행이 **검증 가능(verifiable)** 하려면 세 명제가 보장돼야 한다. (trustless가 아니다 — 검증자 지정과 AI·IoT 신호 생성의 최종 권한이 현재 FarmFi에 있으므로, 외부 오라클·검증인 분산 전까지는 "검증 가능한 집행"으로 표현한다. 운영사 선의에 의존하지 않는 탈출구로 마일스톤 타임아웃 시 누구나 실패 전환을 트리거할 수 있다 — `Escrow.triggerTimeoutFailure`.)

| 명제 | 내용 | 보장 위치 |
|---|---|---|
| **①** | 마일스톤 **미달 시 자금이 풀리지 않음** | `Escrow.releaseTranche`의 `require(milestone.verified)` |
| **②** | **사람의 임의 개입으로 강제 집행 불가** | `verifyMilestone`은 `VERIFIER_ROLE`만, `releaseTranche`는 `verified` 필수 |
| **③** | 마일스톤 **통과 시 추가 승인 없이 자동 집행** | AI 통과 → 서버가 온체인 `verifyMilestone`→`releaseTranche` 자동 호출 |

검증 레이어는 ①의 "미달 판정"과 ③의 "통과 시 자동 집행 트리거"를 담당한다. ②는 컨트랙트 권한 모델이 담당한다.

## 2. 2단계 구조 — 오프체인 판정 → 온체인 집행

```
[증빙 제출] → AI 신호 판정(오프체인) → 신호 AND 집계 → 통과?
                                                    │
                          ┌─── 통과 ───────────────┘
                          ▼
        온체인 verifyMilestone(seq)  ──→  releaseTranche(seq)  ──→  txHash
        (검증 앵커링, 명제 ②)             (트랜치 집행, 명제 ③)

                          └─── 미달 ──→ 해제 없음 + 재검증/수동검토 (명제 ①)
```

- **오프체인(AI)**: 증빙(계약서·영수증·사진·IoT)의 진위·조건부합을 판정해 `passed` 신호를 만든다. 강제 통과 처리 없음 — AI가 거부하면 그대로 차단.
- **온체인(Escrow)**: 통과한 마일스톤만 `verified`로 기록하고 트랜치를 집행한다. 미설정(배포 전)이면 `txHash=null`로 DB 데모만 진행 (`onchain.ts`의 `isOnchainEnabled()`).

## 3. 마일스톤별 검증 명세

총 모집목표 17,500,000원을 4개 트랜치로 분할 집행. (출처: `demo/reset` 시드 = `plan.md` 단위경제)

| seq | 마일스톤 | 해제율 | 해제액(원) | 필요 신호 (AND) | IoT 기간 | 교차검증 |
|---|---|---|---|---|---|---|
| 1 | 공간 준비 | 35% | 6,125,000 | `contract` · `receipt` · `photo` | — | `receipt↔photo` |
| 2 | 시운전 + 안정성 | 30% | 5,250,000 | `iot` | 14일 | — |
| 3 | 첫 수확 + 판매 | 20% | 3,500,000 | `photo` · `receipt` | — | — |
| 4 | 지속 운영 | 15% | 2,625,000 | `iot` · `receipt` | 60일 | — |

- **집계 규칙**: 필요 신호 + 교차검증이 **전부 true**일 때만 마일스톤 통과 (`Object.values(signals).every(v => v === true)`). 하나라도 false면 미달.
- 합계 17,500,000원 = 모집목표 전액 (remaining 0으로 완납).

## 4. 신호별 판정 로직

각 신호는 전용 라우트(`/api/ai/*`)가 `{ passed: boolean, ... }`을 반환한다. AI 추출은 `extractFromImage`(§7) 공용 헬퍼를 쓴다.

### 4.1 `contract` — 임대차 계약서 (`verify-contract`)
계약서에서 임대인·임차인·주소·면적·기간·월세를 추출 후 프로젝트 정보와 대조.

- **추출 성공**: `address` 비어있지 않음 **AND** `areaSqm > 0`
- **주소 일치**: 프로젝트 `location`을 공백 분리한 2글자 이상 토큰 중 하나 이상이 추출 주소에 포함
- **면적 일치**: `|추출면적 − 프로젝트면적| / 프로젝트면적 ≤ 20%`
- **passed** = 추출성공 AND 주소일치 AND 면적일치

### 4.2 `receipt` — 구매/판매 영수증 (`verify-receipt`)
영수증에서 금액·항목·일자 추출 + 마일스톤 조건 부합 판정.

- **추출 성공**: `amount > 0` AND `items.length > 0` AND `date` 문자열 존재
- **조건 부합**: 마일스톤 `conditionText`가 있으면 AI의 `matchesCondition === true` 필요.
  - ⚠️ 프롬프트에 **"IoT 가동률·센서·사진 등 영수증으로 확인 불가한 요건은 별도 신호로 검증되니 판단에서 제외"** 명시 → 영수증은 자기 조건(예: 복수 판매)만 평가. (M4 `conditionText`가 IoT+영수증을 묶어 전달해 영수증 검증기가 IoT까지 떠안던 버그를 분리)
- **passed** = 추출성공 AND 조건부합

### 4.3 `photo` — 현장/수확 사진 (`verify-photo`)
마일스톤 타입별 프롬프트(construction/trial_run/harvest/operation)로 객체·상태 검출.

- Gemini가 객체를 `{label, box_2d}` 형태로 줄 수 있어 **`label` 문자열로 정규화** 후 사용.
- **passed** = `confidence ≥ 0.6` AND `objects.length > 0`

### 4.4 `iot` — 센서 가동률 (`detect-anomaly`)
판정 방식이 마일스톤의 `iotMinDays`에 따라 갈린다.

- **`iotMinDays > 0` (M2=14일, M4=60일)** → **가동률 게이트**: 해당 기간 전체 데이터의 **가동률 ≥ 90%**.
  - 가동률 = 윈도우 내 **"건강한 판독" 비율**. 건강 = 모든 센서가 도메인 정상범위 안 (§5).
  - 빈 윈도우는 가동률 0 → **fail-closed** (데이터 없으면 통과 불가).
- **`iotMinDays = 0`** → **즉시 이상 체크**: 데이터 ≥ 1건 AND 이상 미감지(z-score). 0건 자동 통과 방지.

> 현재 시드 마일스톤에서 `iot` 신호는 M2·M4뿐이고 둘 다 `iotMinDays>0`이라 **가동률 게이트**로 판정된다. `iotMinDays=0` 분기는 폴백.

## 5. IoT 가동률 — 도메인 정상범위 기준 (Z-score / Isolation Forest 미채택)

### 5.1 왜 절대범위인가
가동률 게이트는 "최근 분포 대비 튀었나"(상대)가 아니라 **"도메인상 건강하게 운영됐나"(절대)** 를 물어야 한다.

- **Z-score 맹점**: 평균·표준편차를 *판정 윈도우 자체*에서 뽑으므로, 지속성 고장(예: 히터 고장으로 60일 내내 35℃)이 **새 '정상'으로 흡수**돼 z≈0 → 가동률 100%로 오통과.
- **Isolation Forest 미채택**: IF도 contamination(희귀도) 기반이라 *같은 맹점*을 공유(다수가 된 고장은 안 희귀). 게다가 scikit-learn=Python으로 TS/Vercel serverless 스택에 별도 인프라가 필요한데 데모 정확도 이득이 없음. → **다변량 상관 이상은 향후 확장**으로 두고, 게이트는 도메인 절대범위 규칙으로 구현.
- **Z-score는 유지**: 대시보드의 "방금 이상신호" 보조 표시(단발 스파이크)용으로만 `detectAnomalies`에 남김. 게이트에는 미사용.

### 5.2 정상범위 (새싹삼 재배 도메인)
판독 1건은 **모든 센서가 아래 범위 안일 때만** "건강". (`iot-health.ts`의 `HEALTHY_RANGES`)

| 센서 | 정상범위 | 근거 |
|---|---|---|
| 온도 | 18 ~ 28 ℃ | 생육가능 12~28, 권장 18~25, 25℃↑ 환풍 (진로교육원·RDA) |
| 습도 | 70 ~ 90 % | 새싹 수경 70~90, 재배동 내부 82~88 |
| pH | 5.5 ~ 6.5 | 인삼 토양 적정 pH 5.5~6.5 (농진청/RDA) |
| 조도 | 0 ~ 20,000 lux | 반음지·목표 ~12,000 lux·14h 광주기. 야간 0은 정상이라 **상한만** 게이트 |
| CO₂ | 400 ~ 1,500 ppm | 일반 시설재배 범위(참고값, 새싹삼 전용 출처 없음) |

- **시뮬레이터 정합**: IoT mock(`iot-seed.ts`)은 정상 데이터를 위 범위 안에서 생성하고 2% 비율로 단일 센서를 범위 밖(예: 온도 35℃+, 습도 95%+)으로 주입한다. → 정상 데이터 가동률 ≈ 98% (게이트 90% 통과), 고장 주입분만 차감.
- **한계**: 조도는 "야간 0 정상" 때문에 상한만 검사 → *주간 조명 꺼짐* 고장은 못 잡음(시간대 인지 필요, 향후 확장). 시뮬레이터는 과조도 고장만 주입해 데모 영향 없음.

## 6. 교차검증 (`receipt↔photo`) — M1

영수증 구매 항목과 사진 검출 객체가 **같은 설비 카테고리를 하나 이상 공유**하는지 확인 (서류와 현장의 정합). 4개 카테고리 중 하나라도 양쪽에서 키워드가 잡히면 통과.

| 카테고리 | 키워드 |
|---|---|
| 조명 | led, 조명, 라이트, light, lamp |
| 센서 | 센서, sensor |
| 재배설비 | 재배, 선반, 베드, rack, bed, shelf |
| 관수 | 관수, 급수, 펌프, 양액, pump, irrigation |

`crossCheck` 결과도 `signals`에 포함돼 §3의 AND 집계 대상이 된다.

## 7. AI 추출 스택

- **공용 헬퍼** `extractFromImage` (`ai-vision.ts`): 키가 설정된 provider만 **Gemini → OpenAI → Anthropic** 순으로 폴백. 현재 `GEMINI_API_KEY`만 설정 → 무료 `gemini-3.5-flash`로 동작. 유료 전환은 키만 추가하면 라우트 수정 없이 합류.
- **캐시** `withAICache(milestoneId, type, …)` (`AiCache` 모델): `milestoneId`+신호종류별로 추출 결과 캐싱 → 중복 호출 방지. `demo/reset`은 새 마일스톤 id를 발급하므로 캐시가 자동 무효화됨(단일 스텝 재검증 시엔 해당 캐시 행 삭제 필요).

## 8. 집계 · 재검증 · 온체인 집행 (`milestones/[id]/verify`)

1. `requiredSignals`를 순회하며 각 신호 라우트 호출 → `signals` 수집, `crossCheck` 추가.
2. **게이트**: `signals` 값이 전부 true → `passed`.
3. **통과 시**:
   - 마일스톤 `status = verified`, `aiVerificationResult = signals` 저장.
   - 온체인 `verifyMilestoneOnChain(seq)` 호출(명제 ②). 체인 오류는 try/catch 격리 → DB 데모 무중단. 배포 전이면 `txHash=null`.
   - 트랜치 실제 집행은 후속 `complete` 라우트가 `releaseTrancheOnChain(seq)` 호출(명제 ③).
4. **미달 시** (강제 통과 없음):
   - `retryCount += 1`. **1회 실패** → 재검증 1회 안내 알림. **2회 실패** → `status = manual_review` 전환 + 관리자 알림(미통과 신호 명시).
   - 트랜치 해제 없음 (명제 ①).

## 10. 한계 · 향후 확장

- **조도 주간-암흑 고장**: 시간대 인지 판정 필요 (현재 상한만 게이트).
- **다변량 상관 이상**: 센서 간 상관 붕괴(예: 고온+저습 동시) 탐지는 Isolation Forest 등으로 확장 가능 — 단, 게이트가 아닌 보조 신호 용도.
- **CO₂ 범위**: 새싹삼 전용 근거 확보 시 좁힐 수 있음(현재 일반 시설재배 참고값).
